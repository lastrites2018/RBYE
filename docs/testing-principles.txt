## 테스트 작성 원칙 (결과 중심)

> 실행 절차, 계층별 역할, 운영 체크리스트는 `tests/TESTING_GUIDE.md`를 참고한다.

### 1. 핵심 철학

- 테스트의 목적은 **코드 구현 보호**가 아니라 **동작과 결과 보호**다.
- 검증 기준은 "어떻게 동작했는가"보다 **"예상한 결과가 나왔는가"**다.
- 기준 관점은 내부 로직이 아니라 **사용자 관찰 가능 결과**다.
- 각 테스트의 "사용자"는 **해당 모듈의 직접 소비자**다.
  - 순수 함수의 사용자 = 호출하는 컴포넌트, 훅의 사용자 = 훅을 쓰는 페이지, 페이지의 사용자 = 엔드유저
- 외부 의존(localStorage, fetch, cookie 등)은 **최소 경계에서만 대체**한다.
  - localStorage mock → OK (외부 경계). 내부 setState 호출 여부 assert → 안티패턴 (구현 결합).
- 커버리지는 참고 지표일 뿐이며, 목표는 **중요한 동작의 신뢰성**이다.

---

### 2. 테스트 계층과 한계

각 계층이 잡을 수 있는 버그와 잡을 수 없는 버그가 다르다.
이걸 인지하지 않으면 "테스트를 통과했으니 안전하다"는 착각에 빠진다.

| 계층 | 잡을 수 있는 것 | 잡을 수 없는 것 |
|------|-----------------|-----------------|
| 순수 함수 테스트 | 데이터 변환 오류, 경계값, 로직 실수 | 비동기 타이밍, 컴포넌트 생명주기 |
| 훅 반환값 테스트 | 상태 변화, localStorage 읽기/쓰기 | 마운트 순서, useEffect 실행 타이밍 |
| 통합 파이프라인 테스트 | 함수 조합 시 결과 불일치 | 렌더링 조건 분기, 컴포넌트 간 상호작용 |

**이 프로젝트에서 실제로 발생한 버그 유형:**

- `expandBullets`가 `•` 패턴을 못 잡음 → **순수 함수 테스트로 잡을 수 있었음** (테스트 케이스 부족)
- `BookmarkView` 마운트 시 hydration 전에 `onEmpty` 발동 → **순수 함수 테스트로는 잡을 수 없음** (컴포넌트 생명주기 문제)
- `collapsePreferential` prop이 비동기 로드 후 반영 안 됨 → **순수 함수 테스트로는 잡을 수 없음** (useState 초기값 타이밍)

**교훈:**
개별 모듈이 각각 올바르더라도, **모듈 간 연결 지점의 타이밍**에서 버그가 발생한다.
이건 Hickey가 말한 "composing" 문제 — 부품이 맞다고 조립이 맞는 게 아니다.

### 3. 우리의 전략: 순수 함수 추출 + 접합부 주의

**전략의 핵심:**

1. **가능한 한 많은 로직을 순수 함수로 추출한다.**
   - 순수 함수는 테스트가 쉽고, 조합해도 타이밍 문제가 없다.
   - `jobFilter.ts`, `skillsetMode.ts`, `statsView.ts`, `expandBullets.ts` 등.

2. **순수 함수로 추출할 수 없는 영역(접합부)을 식별하고 주의한다.**
   접합부 = React 생명주기와 외부 상태(localStorage, API)가 만나는 지점.
   - useState 초기값 vs useEffect hydration 타이밍
   - 부모-자식 컴포넌트 간 상태 전파 순서
   - 조건부 마운트/언마운트 시점

3. **접합부에서 발생하는 버그는 패턴으로 방어한다.**
   테스트로 잡기 어려우므로, 코드 패턴 자체로 방지:
   - `mounted` 플래그: hydration 완료 전에 판단하지 않음
   - 콜백 대신 이벤트: 부모-자식 간 직접 의존 줄이기
   - 초기값을 localStorage에서 동기 읽기 (가능한 경우)

---

### 4. 작성 순서

1. **검증할 동작을 한 줄로 정의**
   - 이 함수/컴포넌트/시스템의 목적이 무엇인지 먼저 명확히 한다.
2. **입력과 기대 결과를 정의**
   - 입력 조건과 기대 출력(UI/값/상태 변화)을 짝으로 잡는다.
3. **핵심 시나리오를 우선 테스트**
   - 성공 시나리오, 실패 시나리오, 경계값(엣지 케이스)을 포함한다.
4. **접합부 시나리오를 의식적으로 점검**
   - 새 기능 추가 시 "이 기능은 마운트 타이밍에 의존하는가?" 질문.
   - 의존한다면 mounted 가드 또는 동기 초기화로 방어.
5. **중복 테스트를 제거**
   - 같은 동작을 여러 테스트 계층에서 반복 검증하지 않는다.

---

### 5. 좋은 테스트의 기준

- 테스트 이름만 읽어도 의도와 기대 결과를 이해할 수 있어야 한다.
- 한 테스트는 한 가지 행동과 한 가지 결과에 집중한다.
- 구현 변경(리팩토링) 후 결과가 같다면 테스트가 유지되어야 한다.
- 실행할 때마다 같은 조건에서 같은 결과를 재현할 수 있어야 한다.

---

### 6. 피해야 할 안티 패턴

- 내부 메서드 호출 여부만 검증하는 테스트
- 결과와 무관한 세부 구현에 강하게 결합된 테스트
- 동일 동작을 여러 위치에서 반복 검증하는 중복 테스트
- 커버리지 수치만 올리기 위한 의미 없는 테스트 추가
- **순수 함수 테스트만으로 안전하다고 판단하는 것** ← 추가

---

### 7. 테스트 우선순위

1. 순수 함수 — 입력→출력 명확, 가장 높은 ROI
2. 통합 파이프라인 — 두 함수를 조합했을 때의 결과 검증 (normalizeJobText → expandBullets)
3. 훅 반환값 — 브라우저 API 경계만 대체, 반환값 기준 검증
4. 접합부 점검 — 테스트보다 코드 패턴(mounted 가드)으로 방어

---

### 8. "사용자"의 경계

| 계층                  | "사용자"            | 관찰 가능 결과                          |
|-----------------------|--------------------|-----------------------------------------|
| normalizeJobText      | 호출하는 컴포넌트   | 정규화된 문자열                          |
| useBookmarks          | 훅을 쓰는 컴포넌트   | 반환값 (bookmarks, isBookmarked 등)      |
| BookmarkView          | [type].tsx 페이지   | 즐겨찾기 카드 렌더링 또는 빈 상태        |
| [type].tsx 페이지     | 실제 엔드유저       | 화면에 보이는 공고 목록, 필터 결과        |

---

### 9. 외부 의존 경계 원칙

이 프로젝트는 localStorage, document.cookie, fetch, IntersectionObserver 등
브라우저 API에 강하게 의존한다. 안티패턴(mock 과잉)을 경계하되,
"외부 의존은 최소 경계에서 대체한다"는 원칙을 지킨다.

예시:
- useBookmarks 테스트 시 localStorage 자체를 mock → OK (외부 경계)
- 내부 setBookmarks 호출 여부를 assert → 안티패턴 (구현 결합)
